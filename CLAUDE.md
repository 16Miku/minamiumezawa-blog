# 梅泽美波博客归档项目 - CLAUDE.md

## 核心原则

### 1. 从独特性出发 (First Principles Thinking)

在设计和实现本项目时，剥离所有表面假设，回归技术和需求的本质。

-   **质疑假设**："其他人怎么做"不重要，"梅泽美波的博客数据有什么独特性，如何最优雅地保存和展示"才重要。
-   **需求本质**：我们的目标是构建一个`高度稳定、离线可运行、且视觉体验接近原版`的粉丝归档站。不应为技术而技术。

### 2. 对抗式审查 (Adversarial Review)

作为顶级工程师，对自己写的每一行代码、每一个方案保持怀疑和挑战。

-   **主动寻找缺陷**：在交付代码前，主动思考"如果乃木坂官网改版了怎么办？"、"如果图片下载失败怎么办？"、"如果用户断网了怎么办？"。
-   **压力测试**：不满足于"能运行"，代码必须能处理异常、边界情况和网络抖动。
-   **追求最优而非最快**：拒绝简化的、有潜在风险的方案，寻找最稳健、最优雅的工程实现。

## 项目架构

本项目是**数据驱动的静态站点归档工具**，流水线分为三段：

```
爬取(crawler) → 本地化(image_downloader) → 生成(generator)
```

- `data/raw/blogs.json` 是**数据源真相**，所有页面由它渲染。
- 图片全部本地化到 `data/images/`，零远程依赖（归档的第一性原理）。
- `public/` 是派生产物，不入库，由 `generator.py` 构建时生成。

各脚本职责（详见 README「各脚本职责速查」）：

| 脚本 | 职责 |
|------|------|
| `crawler.py` | 爬列表页+详情页 → blogs.json |
| `fix_dates.py` | 从列表页补全精确日期 |
| `image_downloader.py` | 下载图片 + URL本地化 |
| `cleanup_images.py` | 删除孤儿图片 |
| `generator.py` | Jinja2 渲染静态站 |
| `verify_content.py` | 对照原站校验一致性 |

## 可复用性

**本项目可复用于归档任意乃木坂46成员**——全体成员博客页面结构相同。

成员相关参数当前为**硬编码**，集中在少数几处：
- 成员ID：`crawler.py` / `fix_dates.py` 的 `ARTIST_ID`
- 站点文案：`generator.py` 的标题/简介/年份 + `templates/` 中的成员名

换成员的**完整步骤见 README「复用本项目归档其他乃木坂成员」章节**。
若未来需频繁归档多位成员，建议将硬编码提取到统一 `config.py`（当前为保持稳定未做此重构）。

> 注意：修改成员后，务必先清空 `data/raw/blogs.json`、`data/images/`、`public/`，
> 避免新旧成员数据混淆。
